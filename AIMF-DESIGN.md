# AIMF ŌĆö AI Minimal Format

**Version:** 0.2.0  
**Status:** v2 complete. Compiler, reader, task protocol all implemented.

---

## 1. Problem

LLMs waste tokens on project context. JSON, YAML, and XML carry structural overhead that eats into limited context windows. Loading entire codebases is prohibitive. Loading nothing forces the LLM to ask for files one at a time, burning turns.

AIMF solves this with a navigation manifest ŌĆö a tiny document (~100 tokens) that tells the LLM what exists in a project, what each resource means, how much it costs to load, and when it's relevant. Resources are never embedded. They're loaded on demand from the working folder and prompt-cached.

## 2. Core Design

### The manifest is a map, not a container

An AIMF document describes resources but contains none of them. This is the fundamental design decision. A 500-file project produces the same ~100-token manifest as a 5-file project ŌĆö the @NAV section just has more rows. Token cost scales with what you load, not what exists.

### Three usage scenarios

**Exploration** ŌĆö Load the manifest (~100 tokens). Know the entire project. Fetch individual resources as the conversation progresses. Each loaded resource gets prompt-cached and stays warm across turns.

**Design Mode** ŌĆö Load all resources tagged `design` (typically mermaid diagrams, specs, decision docs). Present a token budget before loading. After approval, the full system architecture is in the prompt cache. Design decisions can reference any part of the system.

**Task Handoff** ŌĆö Opus creates a task document: a curated @NAV subset + @CTX briefing + markdown instructions. Sonnet receives it, loads all listed resources without asking (Opus already curated), and executes the task. The token cost is known upfront.

## 3. Document Format

Two sections, always in this order:

```
@NAV ŌĆö resource catalog
@CTX ŌĆö project state
```

### 3.1 @NAV

Pipe-delimited table. First line is a header. One row per resource.

```
@NAV
id | type | path | about | tokens | load
ARCH | MRM  | docs/architecture.mermaid | System architecture: 4-stage pipeline | 320 | design
D001 | DEC  | decisions/001-use-tauri.md | Use Tauri over Electron for desktop   | 200 | reference
API  | DIR  | src/api/                   | Route handlers, middleware (Rust)     | 1800 | task:*
CFG  | CFG  | Cargo.toml                | Dependencies and build config         | 250 | task:*
```

**Fields:**

| Field | Description |
|-------|-------------|
| id | Short, human-meaningful tag. ARCH, D001, API ŌĆö not F1, F2. Generated by classifier heuristics or manually assigned. |
| type | Resource type. One of: MRM, DEC, DIR, CFG, DOC, SPEC, SRC, TASK. |
| path | Relative to project root. Directories end with `/`. |
| about | One-line description. Max 80 characters. Enough to reason about the resource without loading it. |
| tokens | Estimated token cost to load. Calculated as `ceil(bytes / 4)`. |
| load | When this resource is relevant. Comma-separated hints. |

**Resource types:**

| Type | Meaning | Example |
|------|---------|---------|
| MRM | Mermaid diagram | `architecture.mermaid`, `auth-flow.mmd` |
| DEC | Decision document | `001-use-tauri.md`, ADR-style docs |
| DIR | Source directory (logical unit) | `src/api/`, `lib/core/` |
| CFG | Configuration file | `Cargo.toml`, `package.json` |
| DOC | Documentation | `README.md`, `CONTRIBUTING.md` |
| SPEC | Specification | `AIMF-v2-SPEC.md`, format definitions |
| SRC | Individual source file | `src/main.rs`, entry points |
| TASK | Task document | `TASK-auth.task.md` |

**Load hints:**

| Hint | Meaning |
|------|---------|
| `design` | Load in design mode. Architecture diagrams, specs, decision docs. |
| `task:*` | Load for any task. Configs, source directories, key source files. |
| `task:auth` | Load for auth-related tasks specifically. |
| `reference` | Background context. Load when the user asks about it. |

### 3.2 @CTX

Key-value pairs describing project state. One per line, colon-separated.

```
@CTX
status: v2 complete ŌĆö compiler, reader, task protocol all implemented.
compiled: 2026-04-15
focus: First build and end-to-end testing on local machine.
next: cargo build --features cli, run against this project, test all three modes.
blockers: None.
token_budget: Design mode ~4,455 tokens. Full load ~28,042 tokens.
notes: Compiler requires Rust toolchain. Reader is pure Python (stdlib only).
```

**Common keys:**

| Key | Purpose |
|-----|---------|
| status | Current project state. One sentence. |
| compiled | When this manifest was generated. |
| focus | What area is active right now. |
| next | What should happen next. |
| blockers | Anything blocking progress. |
| token_budget | Estimated costs for different loading modes. |
| repo | GitHub identifier (owner/repo). |
| notes | Anything else relevant to the current session. |

The compiler generates default @CTX entries. Users edit `focus`, `next`, and `blockers` manually to reflect actual project state.

### 3.3 Task Documents

A task document extends the AIMF format with a markdown section separated by `---`:

```
@NAV
id | type | path | about | tokens | load
AUTH | MRM  | docs/auth-flow.mermaid | OAuth2 flow diagram | 480 | design
SRC  | DIR  | src/auth/              | Auth module source  | 1800 | task:auth

@CTX
status: Auth module has session-based login. Adding OAuth2.
focus: OAuth2 provider abstraction.

---
<!-- Total resource cost: ~2280 tokens -->

# Task: Implement OAuth2 Login Flow

Build an OAuth2 provider abstraction in `src/auth/oauth.rs`...
```

The receiving model (typically Sonnet) loads all @NAV resources without asking ŌĆö Opus already curated the minimal set needed for the task. The HTML comment after `---` reports total token cost.

## 4. Compiler Architecture

The compiler is a Rust binary that scans a project directory and produces an `.aimf` manifest. It runs as a 4-stage pipeline.

```
Project Folder ŌåÆ Scanner ŌåÆ Classifier ŌåÆ Curator ŌåÆ Emitter ŌåÆ .aimf
                                Ōåæ
                        Session Miner (optional)
```

### 4.1 Scanner

**Input:** `CompilerConfig` (root path, ignore patterns, include_source_dirs flag)  
**Output:** `Vec<Discovered>` ŌĆö resources worth indexing, sorted by priority

The scanner walks the directory tree and discovers meaningful resources. It is not a full file index ŌĆö it looks for specific patterns:

| What | How detected |
|------|-------------|
| Mermaid diagrams | `.mermaid` or `.mmd` extension |
| Decision docs | Path contains `decision` or `adr`, or filename starts with `0` + `.md` |
| Task docs | Filename ends with `.task.md` or starts with `task-` |
| Specifications | Filename contains `spec` + `.md` extension |
| Key configs | Top-level (depth Ōēż 2): `Cargo.toml`, `package.json`, `pyproject.toml`, `go.mod`, `Gemfile`, `requirements.txt`, `tsconfig.json`, `Dockerfile`, `docker-compose.yml`, `Makefile`, `justfile`, `.env.example` |
| Documentation | `README*`, `CHANGELOG.md`, `CONTRIBUTING.md`, files under `docs/` |
| Source directories | Parent directories (1-2 levels) of recognized source files |

**Ignore patterns** filter entire subtrees by matching any path component. Defaults: `.git`, `node_modules`, `target`, `__pycache__`, `.DS_Store`, `dist`, `build`, `gen`.

**Priority sort** determines @NAV ordering: Mermaid (0) ŌåÆ Specification (1) ŌåÆ Decision (2) ŌåÆ Documentation (3) ŌåÆ Config (4) ŌåÆ SourceDir (5) ŌåÆ SourceFile (6) ŌåÆ Task (7).

**Source directory logic:** For a file at `src/api/handlers/auth.rs`, the scanner extracts the meaningful parent directory at 1-2 levels depth: `src/api/`. This avoids listing every leaf directory while still capturing logical project structure. Source file extensions recognized: `rs`, `py`, `js`, `ts`, `tsx`, `jsx`, `go`, `java`, `kt`, `rb`, `php`, `c`, `cpp`, `h`, `hpp`, `cs`, `swift`, `ex`, `exs`, `vue`, `svelte`, `dart`.

```rust
pub struct Discovered {
    pub path: PathBuf,       // Relative to root
    pub abs_path: PathBuf,   // Absolute path for reading content
    pub is_dir: bool,
    pub size_bytes: u64,     // File size or recursive dir total
    pub kind: DiscoveredKind,
}

pub enum DiscoveredKind {
    Mermaid, Decision, Config, Documentation,
    Specification, SourceDir, SourceFile, Task,
}
```

### 4.2 Classifier

**Input:** Single `Discovered` resource  
**Output:** `(ResourceType, about_string)` ŌĆö type tag and human-readable description

The classifier generates descriptions by reading file content heuristically. No AI calls, no token consumption.

**Type mapping:** `DiscoveredKind` ŌåÆ `ResourceType` is 1:1 (MermaidŌåÆMRM, DecisionŌåÆDEC, ConfigŌåÆCFG, DocumentationŌåÆDOC, SpecificationŌåÆSPEC, SourceDirŌåÆDIR, SourceFileŌåÆSRC, TaskŌåÆTASK).

**Description heuristics:**

| Resource | Strategy |
|----------|----------|
| Mermaid | Extract YAML `title:` from frontmatter, or identify diagram type (graph, flowchart, sequenceDiagram, etc.) |
| Markdown (DEC/DOC/SPEC/TASK) | Extract first `# Heading` or first non-empty, non-frontmatter line |
| Config | Parse `[package] name` from Cargo.toml, `"name"` from package.json, or fall back to filename |
| Source directory | Count files by extension, list subdirectories. e.g., "src/api: handlers, models (Rust)" |
| Source file | Extract module doc comment (`//!`, `///`, `"""`, `'''`) |

All descriptions are truncated to `max_about_len` (default 80 characters) with `...` suffix.

**ID generation:** IDs are short, human-meaningful tags derived from filenames:
- Mermaid: abbreviate filename ŌåÆ `architecture.mermaid` becomes `ARCH`
- Decision: leading digits ŌåÆ `001-use-tauri.md` becomes `D001`
- Source directory: abbreviate last component ŌåÆ `src/api/` becomes `API`
- Config/Doc: abbreviate filename ŌåÆ `Cargo.toml` becomes `CARG`, `README.md` becomes `READ`

**Load hint defaults:**

| Type | Default hints |
|------|--------------|
| MRM | `design` |
| DEC | `reference` |
| DIR | `task:*` |
| CFG | `task:*` |
| DOC | `reference` |
| SPEC | `design`, `reference` |
| SRC | `task:*` |
| TASK | `task:*` |

### 4.3 Curator

**Input:** `Vec<Discovered>`, `CompilerConfig`, `HashMap<String, String>` (session enrichments)  
**Output:** `(Vec<NavEntry>, Vec<CtxEntry>)` ŌĆö the structured AIMF document

The curator is the orchestrating stage. For each discovered resource, it:

1. Calls `classifier::classify()` to get type and description
2. Checks session enrichments ŌĆö if a richer description exists for this path, uses it instead
3. Generates a unique ID via `classifier::generate_id()`, appending numeric suffixes for collisions (ARCH, ARCH2, ARCH3)
4. Estimates token cost: `ceil(bytes / 4.0)` ŌĆö a heuristic that maps roughly to GPT/Claude tokenization
5. Assigns load hints via `classifier::default_load_hints()`
6. Appends `/` to directory paths

After processing all resources, the curator generates default @CTX entries:

| Key | Generated value |
|-----|----------------|
| status | "Project indexed: N resources (X diagrams, Y decisions, Z source dirs)." |
| compiled | "YYYY-MM-DD ŌĆö auto-generated by aimf compiler." |
| token_budget | "Full load: ~N tokens. Design mode: ~M tokens." |
| repo | GitHub identifier if provided in config |
| focus | "[edit this: what area is active right now]" |
| next | "[edit this: what should happen next]" |

```rust
pub struct NavEntry {
    pub id: String,
    pub resource_type: ResourceType,
    pub path: PathBuf,
    pub about: String,
    pub tokens: usize,
    pub load_hints: Vec<String>,
}

pub struct CtxEntry {
    pub key: String,
    pub value: String,
}
```

### 4.4 Emitter

**Input:** `AimfDocument` (nav + ctx vectors)  
**Output:** Formatted AIMF text string

Two output modes:

**`emit(doc) -> String`** ŌĆö Standard manifest. Writes @NAV with column-aligned pipes and a header row, then @CTX with simple key-value lines. Column widths are calculated dynamically from the longest entry in each field.

**`emit_task(nav, ctx, title, body) -> String`** ŌĆö Task handoff document. Same @NAV and @CTX formatting, followed by `---` separator, a token cost comment, and the markdown task body.

### 4.5 Session Miner (Optional)

**Input:** Path to Claude Code session directory (`.claude/projects/<id>/`)  
**Output:** `HashMap<String, String>` ŌĆö path ŌåÆ enriched description

The session miner parses `.jsonl` Claude Code session logs to extract richer resource descriptions without AI calls. It:

1. Finds all `.jsonl` files in the session directory
2. Regex-extracts mermaid code blocks (handles both `\n` and `\\n` escapes from JSONL)
3. For each mermaid block: extracts diagram type, YAML title, and nearby file path references (┬▒500 chars context window)
4. Maps mentioned `.mermaid`/`.mmd` file paths to extracted titles/types
5. Returns enrichments that override classifier descriptions when they're more specific

File path extraction regex recognizes: `.rs`, `.ts`, `.tsx`, `.js`, `.py`, `.toml`, `.json`, `.md`, `.mermaid`, `.mmd`, `.sql`, `.yaml`, `.yml`.

## 5. Public API

The compiler exposes three functions from `lib.rs`:

```rust
/// Full pipeline: scan ŌåÆ classify ŌåÆ curate ŌåÆ emit. Returns AIMF text.
pub fn compile(config: &CompilerConfig) -> String

/// Same pipeline but returns structured document for programmatic use.
pub fn compile_to_doc(config: &CompilerConfig) -> AimfDocument

/// Compile, then filter @NAV to listed resource IDs and emit as task document.
/// Pass "*" in resource_ids to include all resources.
pub fn generate_task(
    config: &CompilerConfig,
    task_title: &str,
    task_body: &str,
    resource_ids: &[String],
) -> String
```

### Configuration

```rust
pub struct CompilerConfig {
    pub root: PathBuf,              // Project root directory
    pub repo: Option<String>,       // GitHub "owner/repo"
    pub branch: Option<String>,     // Git branch name
    pub session_logs: Option<PathBuf>, // Path to .claude/projects/<id>/
    pub ignore_patterns: Vec<String>,  // Directory names to skip
    pub include_source_dirs: bool,     // Add DIR entries (default: true)
    pub max_about_len: usize,          // Description truncation (default: 80)
}
```

Default ignore patterns: `.git`, `node_modules`, `target`, `__pycache__`, `.DS_Store`, `dist`, `build`, `gen`.

## 6. CLI

Feature-gated behind `--features cli`. Uses clap v4.

```
aimf [OPTIONS]

Options:
  -p, --path <PATH>        Project root directory [default: .]
  -r, --repo <REPO>        GitHub identifier (owner/repo)
  -b, --branch <BRANCH>    Git branch name
  -s, --sessions <PATH>    Claude Code session logs directory
  -o, --output <FILE>      Output file (default: stdout)
      --include-dirs        Include source directory entries [default: true]
```

**Examples:**

```bash
# Compile current directory, print to stdout
aimf

# Compile a project to a file
aimf --path ./myproject --output project.aimf

# With GitHub metadata and session enrichment
aimf --path . --repo owner/repo --branch main \
     --sessions ~/.claude/projects/myproject/ \
     --output project.aimf
```

## 8. Reader (Python)

### Parser

Pure Python, no dependencies. Located at `aimf-reader/scripts/parse_aimf.py`.

```bash
python parse_aimf.py project.aimf           # Full JSON summary
python parse_aimf.py project.aimf --nav      # @NAV entries as JSON
python parse_aimf.py project.aimf --ctx      # @CTX entries as JSON
python parse_aimf.py project.aimf --budget design  # Token budget for design mode
python parse_aimf.py project.aimf --search auth    # Search by id/path/about
python parse_aimf.py project.aimf --load ARCH      # Load resource content by ID
```

### Cowork Skill

The reader skill (`aimf-reader/SKILL.md`) defines how an LLM should interact with AIMF documents. Three operating modes:

**Exploration Mode (default):** Read @NAV and @CTX. Summarize what's available. Wait for the user to ask about specific areas. Load resources on demand ŌĆö each one gets prompt-cached.

**Design Mode:** Filter @NAV for `load: design` entries. Present a token budget table showing each resource and its cost. Wait for user approval. Load approved resources. All loaded resources are prompt-cached ŌĆö ready for cross-system design work.

**Task Execution Mode:** The document includes @NAV + @CTX + markdown instructions (after `---`). Load ALL @NAV resources without asking ŌĆö Opus already curated the minimal set. Read task instructions and execute.

## 9. Token Economics

AIMF's value proposition is measurable in tokens:

| What | Tokens | When |
|------|--------|------|
| Manifest itself | ~100 | Always loaded |
| Design mode (diagrams + specs) | ~3,000ŌĆō5,000 | When designing |
| Single resource on demand | varies by file | When needed |
| Full project load | varies | Rarely needed |

The `bytes / 4` heuristic for token estimation is deliberately simple. It consistently overestimates by ~10-20%, which is the safe direction ŌĆö better to budget high than run out of context.

Prompt caching makes the manifest nearly free after the first turn. Resources loaded on demand also get cached and stay warm across turns (5-minute TTL on Anthropic's API).

## 10. Formal Grammar (EBNF)

```ebnf
AIMFDocument   = NavSection, CtxSection, [ TaskSection ] ;

NavSection     = "@NAV", NL, HeaderRow, { NavRow } ;
HeaderRow      = "id", SEP, "type", SEP, "path", SEP, "about", SEP, "tokens", SEP, "load", NL ;
NavRow         = ID, SEP, Type, SEP, Path, SEP, About, SEP, Tokens, SEP, LoadHints, NL ;

CtxSection     = "@CTX", NL, { CtxLine } ;
CtxLine        = Key, ":", SP, Value, NL ;

TaskSection    = "---", NL, [ Comment, NL ], TaskBody ;
Comment        = "<!--", TEXT, "-->" ;
TaskBody       = { ANY } ;

(* Lexical *)
SEP            = SP, "|", SP ;
ID             = 1*( ALPHA | DIGIT ) ;
Type           = "MRM" | "DEC" | "DIR" | "CFG" | "DOC" | "SPEC" | "SRC" | "TASK" ;
Path           = 1*( VCHAR - "|" ) ;
About          = 1*( VCHAR - "|" ) ;         (* max 80 chars *)
Tokens         = 1*DIGIT ;
LoadHints      = LoadHint, { ",", SP, LoadHint } ;
LoadHint       = "design" | "reference" | "task:", 1*( VCHAR - "|" - "," ) ;
Key            = 1*( ALPHA | "_" ) ;
Value          = 1*( VCHAR | SP ) ;

ALPHA          = "A"-"Z" | "a"-"z" ;
DIGIT          = "0"-"9" ;
VCHAR          = %x21-7E ;
SP             = " " ;
NL             = %x0A ;
ANY            = %x00-10FFFF ;
```

## 11. Build & Run

All Rust work happens inside `aimf-compiler/src-tauri/`.

```bash
# Library only (no binary)
cargo build

# CLI binary
cargo build --features cli
cargo run --features cli --bin aimf -- --path /path/to/project

# Tauri desktop app
cargo build --features tauri

# Checks
cargo check
cargo clippy
```

**Dependencies:** serde 1, serde_json 1, walkdir 2, chrono 0.4, regex 1. Optional: clap 4 (CLI), tauri 2 (desktop).

**Python reader:** No dependencies. Requires Python 3.6+.

```bash
python aimf-reader/scripts/parse_aimf.py project.aimf
```
